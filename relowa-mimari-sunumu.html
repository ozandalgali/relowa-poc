Relowa Platformu Mimari Yaklaşımı

Bu doküman, ilk teknik beyin fırtınamızdan çıkan yaklaşımın detaylı bir açıklamasıdır. Mail'deki özetin arkasındaki gerekçeleri, trade-off'ları ve karar noktalarını burada açıyorum — sohbette derinleşmek istediğimiz her başlık burada referans olarak duruyor.

01. Yaklaşımın Özeti

Üç prensip her kararı yönetiyor: substrate sıkıcı ve sağlam, modüller bağımsız evrilebilir, KVKK pozisyonu ilk günden hukuki olarak savunulabilir.

Senin koyduğun çerçeve çok netti — Klasik MVP-sonra-rewrite mantığı yok, her faz production-ready ve kalıcı. Buna bağlı kaldım. Mimaride üç katman var:

  1. Substrate (temel): Postgres + AWS yönetilen servisleri + Supabase açık kaynak bileşenleri. Bu katman değişmiyor; üzerinde modüller geliyor.
  2. Modüller (Lego parçaları): Marketplace (ihale akışı), AI Analiz, Lojistik & Filo, ESG Raporlama, Escrow Ödeme. Her biri EventBridge omurgası üzerinden bağımsız konuşuyor.
  3. Frontend: Next.js (web + PWA), gerekirse Phase 2'de carrier için React Native.

Bu yapı senin "her faz production-ready, sonraki faz Lego parçası" çerçeveni mühendislik diline çeviriyor.

"Klasik 'hızlıca bir MVP yapalım, sonra kodu çöpe atıp baştan yazarız' mantığında ilerlemeyeceğiz. Ürünü faz faz, tamamen modüler bir yapıda geliştireceğiz. Ancak canlıya alacağımız her bir faz/modül, geçici bir MVP değil, projenin 'son noktasıymış' gibi production-ready kalitede olacak."

03. Teknoloji Yığını ve Gerekçeleri

Her teknoloji seçimi bir trade-off'la geliyor. Burada sadece "ne kullandığımı" değil, "neyi reddettiğimi ve niye" yi açıyorum — çünkü bu projede yanlış aracı seçmek yanlış aracı kullanmaktan daha pahalıya patlar.

Veri Katmanı: PostgreSQL (RDS)

Neden: İlişkisel veri (organizasyon → kullanıcı → ihale → teklif → sevkiyat → ödeme) Postgres'in tam olarak en iyi olduğu alan. Multi-tenant izolasyon için Row-Level Security (RLS) native — kural veritabanının içinde, uygulama bug'ı bypass edemez. Time-series veri (piyasa fiyat endeksi, GPS log'ları) için TimescaleDB extension'ı, coğrafi sorgular için PostGIS extension'ı aynı veritabanında çalışıyor. Ayrı bir time-series DB veya GIS DB gerekmiyor.

RDS Multi-AZ ne demek: Aynı verinin Frankfurt'ta iki farklı binada (availability zone) eşzamanlı kopyası tutuluyor. Bir bina kapansa bile diğeri otomatik devralıyor (~30 saniye downtime). Backup, point-in-time recovery, otomatik patching hepsi AWS sorumluluğunda — biz uygulama kodunu yazıyoruz.

Uygulama Servisleri: Supabase OSS Bileşenleri

Supabase Cloud'a bağlı kalmıyoruz, açık kaynak bileşenlerini kendi AWS hesabımızda çalıştırıyoruz. Şu beş bileşen:

  • GoTrue — Auth servisi (login, signup, JWT, email doğrulama)
  • PostgREST — Postgres'i otomatik REST API'ye çeviriyor
  • Realtime — WebSocket katmanı (Postgres değişikliklerini canlı UI'ye push)
  • Storage API — S3'e bağlı dosya yönetimi (RLS-aware access control)
  • Studio — Admin UI (sadece iç ekip için, public değil)

Niye Supabase OSS

  • Vendor lock-in yok — vendor biziz, bileşenler kendi AWS hesabımızda
  • Veri tamamen bizim VPC'mizde, KVKK posture clean
  • Phase 1'de 6-8 hafta tasarruf — auth, REST, realtime sıfırdan yazmıyoruz
  • Her bileşen bağımsız değiştirilebilir — auth'u Cognito'ya, REST'i NestJS'e taşımak Phase 3'te mümkün, schema yerinde durur
  • Postgres altta, Supabase üstte ince katman — geri çekildiğinde sadece Postgres kalır, kod kaybı yok

Niye Supabase OSS Değil (alternatif)

  • Operasyonel basitlik isteniyorsa Supabase Cloud EU region (Frankfurt) — zaten AWS üzerinde, KVKK için DPA imzalanabiliyor, SOC 2 + ISO 27001 sertifikalı
  • Phase 1'de Cloud, Phase 2'de self-host'a migration — schema ve kod migrasyonda değişmiyor
  • Tamamen AWS-native istiyorsak: Cognito (auth) + Lambda (API) + AppSync (realtime) — ama bu ~2 ay ek geliştirme
  • "Hand-rolled NestJS + custom auth" — full kontrol, ama solo lead için Phase 1 takvimini patlatır

Container Orkestrasyon: ECS Fargate

Konteyner çalıştırma katmanı. Fargate "sunucusuz konteyner" demek — biz "şu Docker image'ı 1 kopya çalıştır, 0.5 vCPU + 1 GB RAM ver" diyoruz, AWS arkasında EC2'ları yönetiyor, biz görmüyoruz. Konteyner çökerse Fargate yeniden başlatıyor.

Entegrasyon Omurgası: AWS EventBridge

Niye Phase 1'de gerekli: Senin "Lego parçası" çerçeven tam olarak bu. Modüller birbirine doğrudan bağımlı olmak yerine event'ler üzerinden konuşuyor. Phase 1'de tender.created event'i atılıyor, Phase 2'de matching servisi geldiğinde bu event'i dinlemeye başlıyor — Phase 1 koduna dokunmadan. Phase 3'te IoT modülü geldiğinde aynı şekilde.

EventBridge yerine Kafka / MSK yapsak overkill — pilot ölçekte EventBridge yeterli, Kafka'ya geçiş Phase 4+'da yapılır eğer event log replay gerektiren karmaşık stream processing ihtiyacı çıkarsa.

Frontend: Next.js + PWA

Üretici, tesis, admin ekranları tek codebase. App Router + TypeScript + Tailwind + shadcn/ui (Radix altyapısı), i18n (TR/EN), Zod form validation, Zustand local state, TanStack Query server state. PWA olarak çalışıyor — "ekrana ekle" + push notifications + offline tolerance, ama mobile app store'a girme yükü yok.

Carrier (sürücü) tarafı için ayrı karar: Eğer Phase 1'de carrier sürücü aktif kullanıcıysa (canlı GPS yayını, background tracking, native push), iOS'un PWA kısıtlamaları yüzünden React Native (Expo) ile minimal mobil app gerekiyor. Eğer carrier elle yönetilen bir hizmetse Phase 1'de PWA yeterli, mobil app Phase 2'ye kalır. Bu ~2 ay'lık scope farkı.

AI / Görsel İşleme: FastAPI

İhale yayın akışında foto analizi (saflık, kontaminasyon, kalite skoru) Phase 1 critical path. FastAPI ECS Fargate üzerinde ayrı bir servis olarak duruyor; auto-generated OpenAPI ile dokümantasyon kendiliğinden geliyor. Phase 1'de Greyparrot commercial API entegrasyonu en hızlı yol — KVKK ve ekonomi gerektirirse Phase 2'de self-hosted vision modele geçiş.

Observability (Gözlem)

  • Sentry (EU region) — uygulama hata takibi, source map'li stack trace, release health
  • PostHog (self-hosted) — ürün analitiği, funnel'lar, feature adoption — KVKK için kendi VPC'mizde
  • CloudWatch + CloudTrail — AWS infrastructure metric'leri ve API audit log'ları
  • Audit trail (uygulama-level) — Postgres'te append-only audit_events tablosu + S3 Object Lock mirror; bu observability değil, yasal kanıt katmanı, ayrı tutuyoruz

04. Supabase Tartışması

Bu seçim haklı sorulara açık. Bu yüzden bu bölüm sadece "Supabase iyidir" demiyor — endişeleri tek tek karşılıyor.

Endişe 1: "Supabase MVP-vari değil mi? Ciddi bir proje için zayıf kalmaz mı?"

Burada iki ayrı şeyi karıştırmamak lazım:

  • Yazdığımız kod: Schema, RLS politikaları, business logic, audit altyapısı, migration disiplini, test suite. Bunların hiçbiri Phase 1'de yaz Phase 3'te at değil. Year-1 schema'sı year-5'te de yaşıyor. Bu senin "production-ready, atılmayacak" ilkenle birebir.
  • Substrate bileşenleri: Postgres, GoTrue, PostgREST, Realtime, Storage. Bunlar infrastructure parçaları — bir SaaS'in yaşam döngüsünde infrastructure parçaları zaten zamanla swap edilir. Bu rewrite değil, healthy evolution. Notion ilk yıllarında tek Postgres'teydi, sonra sharding ekledi. GitHub MySQL'le başladı, Vitess ekledi. Bu "kod atma" değil.

Yani Supabase OSS bizi "ciddi olamayan bir noktaya" hapsetmiyor. Tam tersine — her bileşen swappable, sadece doğru zamanda swap ediyoruz. Bu senin Lego çerçevenin tam olarak aradığı şey.

Endişe 2: "Supabase'in RBAC'i zayıf değil mi? Org / role hiyerarşisini nasıl yapacağız?"

Burada haklı bir gözlem ama yanlış sonuç çıkarılabiliyor. Supabase Auth'un "built-in RBAC"i basit case için yeterli, complex B2B case için biz kendi modelimizi yazıyoruz — ki bu Supabase'in eksikliği değil, B2B SaaS'in doğası. Aynı şeyi Cognito ya da Auth0 ile yapsak da bu modeli sıfırdan yazmamız gerekirdi.

Pratikte 4 tablo + 2 SQL helper function + bir kaç RLS policy:

organizations (id, type, name, vergi_no, address, ...)
users (auth.users'ı extend eder, profile bilgileri)
org_members (org_id, user_id, role)
org_invitations (token, org_id, email, role, expires_at)

current_org_id() → JWT'den aktif org çek
user_has_role(role) → bu org'da bu rol var mı kontrolü

RLS policy örneği:
  CREATE POLICY tenant_isolation ON tenders
    USING (org_id = current_org_id());

Bu pattern dünya çapında her B2B SaaS'in yaptığı şey. Postgres + RLS aslında Cognito + Lambda triggers veya Auth0 Organizations'tan daha temiz bir cevap, çünkü kural veri katmanında — uygulama bug'ı bypass edemiyor.

Endişe 4: "PWA, mobil uyumluluk?"

Backend (Supabase + RDS + EventBridge + S3) frontend'in ne olduğunu umursamıyor — REST + WebSocket + Storage API her şeyle konuşuyor. Yani:

  • Web + PWA — Supabase ile sorunsuz çalışıyor, Realtime WebSocket'leri PWA'da çalışıyor, GoTrue token refresh PWA'da çalışıyor
  • React Native (Expo) — Supabase'in resmi React Native SDK'sı var, sıfır sürtünmeyle entegre
  • Capacitor (web → native paketleme) — Yine sorunsuz

Frontend kararı backend kararını değiştirmiyor. Phase 1'de PWA, Phase 2'de carrier için React Native ekleyebiliriz — backend'e dokunmadan.

Postgres katmanı leader-follower replication ile milyonlarca kullanıcıya kadar gider (GitHub yıllarca tek-leader Postgres'te çalıştı, Notion 2022'ye kadar). Supabase services katmanı stateless — task sayısını artırmak yetiyor. Gerçek "scaling wall" Realtime'ın 100K+ concurrent connection bölgesinde geliyor; bu da Relowa için en az 3-4 yıl sonrası, ve geldiğinde Realtime'ı AppSync ya da custom Elixir cluster'la swap etme planı zaten elimizde — schema değişmiyor, kod değişmiyor.

05. Plan B — Supabase OSS Yerine Hand-Rolled Stack

Eğer Supabase OSS yolu (operasyonel, stratejik veya başka bir nedenle) tercih edilmezse hazır bir yedek planımız var. Bu plan "panik kararı" değil — olgun, kanıtlanmış parçalardan kurulu bir alternatif. Burada anahtar prensip: Postgres ve schema yerinde duruyor, sadece üst katmanı manuel kuruyoruz.

Niye bu plan önemli: Supabase OSS'in gerçekten bir limit dayatabileceği iki senaryo var: (1) çok özel bir auth gereksinimi (örn. zorunlu offline-capable mobile auth, veya çok özel SSO compliance), (2) Realtime'ın 100K+ concurrent connection'a çıkması (yıllar sonra). İkisi de Phase 1'de gerçek değil ama "kapı açık" olduğunu göstermek istiyorum. Plan B'nin maliyeti: Phase 1 takvimi 4 ay yerine ~5.5 ay. Her parçası kanıtlanmış, hiçbiri exotic değil.

Plan B Bileşenlerinin Detayı

Drizzle ORM (PostgREST yerine)

2023-2024'te yükselen modern TypeScript ORM. Prisma'dan daha hafif, SQL'e daha yakın, RLS ile düzgün çalışıyor (Prisma'nın bilinen bir zayıflığı RLS'i unutması). Migration disiplini güçlü, schema değişiklikleri code-review'lanabilir.

Niye Drizzle, Prisma değil: Prisma generated client şişman, RLS tarafında nüans var, ve query builder bazen optimal-altı SQL üretiyor. Drizzle daha ince, SQL'e yakın ve raw SQL escape hatch'i temiz.

Better-Auth (GoTrue yerine)

2024-2025'te en hızlı büyüyen auth kütüphanesi. Açık kaynak, Postgres native, Drizzle ile first-class entegrasyon. Email/password, OAuth (Google/Microsoft/GitHub), magic links, MFA, passkeys, organizations, SAML SSO — hepsi içinde. Vercel'in Auth.js'inin modern alternatifi.

Niye Better-Auth, NextAuth değil: NextAuth (Auth.js v5) iyi ama modular değil ve bazı edge case'lerde takılıyor. Better-Auth tamamen modular, plugin sistemli ve TypeScript-first.

Hono (PostgREST yerine API katmanı)

Express'in modern hâli — minimalist, hızlı, edge-runtime uyumlu, Zod entegrasyonu first-class, OpenAPI auto-generation built-in. RPC mode'unda type-safe client export ediyor (tRPC benzeri ama OpenAPI-uyumlu).

NestJS alternatifi: Eğer enterprise-y dependency injection + module sistemi tercih edilirse NestJS de iyi. Ama solo lead için Hono daha az boilerplate.

AWS AppSync (Supabase Realtime yerine)

AWS'in managed GraphQL + WebSocket subscription servisi. KVKK uyumlu, Frankfurt'ta var, AWS-native. Postgres CDC (change data capture) yerine event-driven subscription'lar — bid geldiğinde dashboard'a push, auction state değişince UI güncellemesi.

Pusher alternatifi: AppSync biraz karmaşıksa Pusher (managed) ya da Soketi (open source self-hosted) ile başlanabilir, sonra geçilir.

S3 + Presigned URL (Supabase Storage yerine)

Direkt S3 kullanıyoruz, üzerine kendi access-control handler'ımızı yazıyoruz. Upload için presigned URL üretiyor (RLS'e benzer policy check'i Hono endpoint'inde), download için signed URL veriyor. ImgProxy container'ı resize/optimization için (open source). Audit için S3 Object Lock zaten Plan A'da da vardı.

Trade-off'lar Açıkça

Plan B'nin Avantajları

  • Tam kontrol — her bileşeni biz seçiyoruz, tek başına değiştirilebilir
  • Tek vendor algısı yok — "Supabase'e bağımlıyız" iddiası ortadan kalkıyor
  • Modern, developer-experience odaklı parçalar — Drizzle, Better-Auth, Hono hepsi 2024-2025'in en sevilen araçları
  • Type safety end-to-end — Drizzle'ın schema → TypeScript inferencing'i + Hono RPC mode = client-server type sharing
  • Acquirer due diligence'ında "her şey kendimiz yazdık" hikayesi (eğer önemliyse)

Plan B'nin Dezavantajları

  • ~6-8 hafta ek geliştirme — Phase 1 takvimi 4 ay yerine ~5.5 ay
  • Daha fazla integration glue kodu — Better-Auth ↔ Drizzle ↔ Hono'yu birbirine bağlamak el işi
  • Admin UI yok hazır — Supabase Studio'nun yerine geliştirme/debug için TablePlus + Drizzle Studio (yeterli ama Supabase Studio kadar zengin değil)
  • Realtime patterns daha manuel — AppSync'i Postgres CDC ile bağlamak ya da event-driven kurmak ekstra iş
  • Storage RLS-uyumlu access control'ünü biz yazıyoruz — Supabase'in built-in storage policy'si yerine
  • "Eski" değil ama daha az test edilmiş kombinasyon — Drizzle + Better-Auth + Hono kombinasyonu yeni; her parça sağlam ama birlikte yaşadığı kilometresi az

Dezavantajı: Clerk per-MAU ücretli (~$25/ay'dan başlayıp scale ile artıyor), WorkOS SSO başına ücretli ama enterprise sales'te değer veriyor.

06. KVKK & Veri Yerleşimi

Atık verisi + ihale verisi + escrow ödeme verisi ciddi bir KVKK posture gerektiriyor. Burada hukuki ve mühendislik tarafını net ayırıyorum.

Mevcut Durum: AWS Türkiye'de Ne Var, Ne Yok

Konum türüTürkiye'de var mı?Pratik anlamı
AWS Region (full region: RDS, S3, EventBridge, hepsi) Yok En yakın region: eu-central-1 Frankfurt
AWS Local Zone (compute uzantısı, sınırlı servis) Var (eu-central-1-ist-1, Mayıs 2025) EC2, EBS, ALB var; RDS, S3, Lambda, EventBridge YOK — parent region'da kalıyor
AWS Direct Connect (network bağlantısı) Var (Equinix IL4, Mayıs 2025) Sadece bir backbone bağlantısı; veri yine Frankfurt'ta tutuluyor
CloudFront edge (CDN cache) Var (Şubat 2024'ten beri) Sadece statik asset cache, dynamic compute yok
Kritik nokta: Direct Connect ve Local Zone duyuruları çok dolaşıyor ama "AWS Türkiye'de region açtı" anlamına gelmiyor. Local Zone parent region'a bağlı bir uzantı, full region değil. RDS muhtemelen yok, S3 muhtemelen yok, EventBridge muhtemelen yok. "Tüm sistem İstanbul'da" demek için bu servislerin parent region'da kalmasıyla yaşamak gerekir, ki bu pratikte hibrit bir yapı demek.

Önerilen Strateji: Hukuki + Mühendislik İki Katmanlı

Hukuki Katman (DPA + SCC + Frankfurt)

KVKK'da kişisel verinin yurt dışı aktarımı için iki temel araç var:

  • DPA (Data Processing Agreement): Müşteri ile imzaladığımız "verinizi şu şekilde işliyoruz, taahhüt ediyoruz" sözleşmesi. Müşteri = veri sorumlusu, Relowa = veri işleyen.
  • SCC (Standard Contractual Clauses): Relowa ↔ AWS arasındaki "veriyi yurt dışına aktardığımda AWS de aynı koruma standartlarını uyguluyor" sözleşmesi. KVKK Kurulu'nun kabul ettiği şablon.

İkisi birlikte "veri Frankfurt'ta ama hukuki olarak korumalı" pozisyonu. Türk B2B SaaS'lerinde yaygın ve kabul görmüş bir pratik. Sektörel düzenlemelerin (Çevre Bakanlığı yönergeleri, kurumsal müşterilerin internal compliance) ne istediği ayrı bir konu — bunun için bir KVKK danışmanı veya avukat netleştirmeli.

Mühendislik Katmanı (Benim Sorumluluk Alanım)

  • Encryption-at-rest: RDS encryption (KMS), S3 encryption, EBS encryption
  • Encryption-in-transit: TLS 1.3 her yerde, internal traffic dahil
  • Role-based access control: Multi-tenant RLS + IAM roles, least-privilege
  • Audit trail: Append-only audit_events + S3 Object Lock (WORM, değiştirilemez)
  • Data retention policy: Hangi veri ne kadar saklanıyor, kullanıcı silme hakkı
  • Veri ihlali müdahale planı: 72 saat bildirim hazırlığı (KVKK gereksinimi)

Phase 2 Evrim Yolu

Phase 2'de İstanbul Local Zone'un servis matrisi olgunlaştığında (özellikle RDS ve Fargate desteklendiğinde), hot path bileşenlerini İstanbul'a taşıyabiliriz. Hot path = kullanıcının her tıklamada hissettiği interactive akış (login, ihale oluştur, canlı bid). Cold path = arka plan işleri (PDF üretim, audit mirror, backup) Frankfurt'ta kalıyor — latency'leri kritik değil, maliyet ve dayanıklılık önemli.

Bu modüler mimarinin tam olarak söz verdiği şey — kod değişmiyor, sadece infrastructure provisioning değişiyor.

07. Veri Akışı & Sistem Mimarisi

Önce kelimeyle özetliyorum, sonra dört diyagramla görsel açılım yapıyorum. Her diyagrama altında hangi DDIA prensibinin yerleştiğini etiketliyorum.

Özet

Multi-tenant veri modeli

Müşteri = organizasyon (Producer / Recycler / Carrier type'lı). Her org'da birden fazla kullanıcı, kullanıcı başına intra-org rol (Full Admin, Muhasebe, Operasyon Yöneticisi — Yetkili Yönetimi ekranında görünen). Üç temel tablo: organizations, users, org_members (org_id + user_id + role). Org type'a özel bilgiler organizasyon seviyesinde tutulur (org_profile_producer, org_profile_recycler, org_profile_carrier) — kullanıcı seviyesinde değil. Bir kullanıcı teknik olarak birden fazla org'a üye olabilir; aktif org JWT claim'inde (active_org_id) taşınır, switch ettiğinde yeni JWT issue ediliyor.

Talep oluşturma akışı (özet)

Kullanıcı login olur (GoTrue → JWT, içinde user_id + active_org_id + role claim'leri). Next.js server action input'u Zod ile doğrular. Postgres'e insert atılır — RLS politikası org_id + role'ü doğrular, tek transaction içinde audit_events tablosuna da append edilir (insert-only, hash-chained, update/delete RLS ile kapalı). Commit'ten sonra DB trigger EventBridge event'i yayar: tender.created. Bu event'i üç yer dinler: Realtime kanalı eşleşen tesislerin WebSocket'ine push atar (anlık UI), SES mail bildirimi gönderir, ve Phase 2 seam olarak matching servisi event'i tüketip akıllı ihaleyi başlatır. Tesis dashboard'unu açtığında PostgREST sorgusu otomatik filtreler (RLS: org_id + bölge + materyal tipi), yeni talebi anında görür, teklif gönderir, mirrored akış geri döner. Carrier kullanıcısı (Phase 1'de scope'a girerse) kendi rolüne göre sadece atanmış sevkiyatları görür — aynı RLS yapısı, farklı policy.

Vurgulamak istediğim dört prensip

  • RLS ile çift-katmanlı izolasyon — org membership + intra-org role. Kural veritabanının içinde, uygulama bug'ı bu sınırı kıramıyor. Tüm policy'ler ortak current_org_id() ve user_has_role_in_org() helper'ı üzerinden yazılıyor; CI'da explicit RLS policy test suite ile cross-tenant data leak baştan testleniyor.
  • Her mutasyonda idempotency key — network retry'ları çift kayıt üretmiyor; escrow Phase 1'e girince absolute zorunluluk hâline geliyor.
  • Append-only audit log + EventBridge — hem yasal gereklilik (KVKK + landing page'deki "değiştirilemez veri" sözü), hem Phase 2/3 modüllerinin entegrasyon noktası. Aynı yapı iki amaca hizmet ediyor.
  • Auction kapanması server-authoritative — kalan süre client'ta render ediliyor ama gerçek kapanma EventBridge Scheduler ile server-side tetikleniyor. Para'ya dokunan akışta client saatine güvenmiyoruz; last-minute sniping'e karşı standart soft-close pattern'i (son saniyelerde gelen bid süreyi uzatır).

Diyagram 1 — Sistem Topolojisi

Tüm bileşenler ve aralarındaki bağlantılar. Her ok kimin kimi çağırdığını gösteriyor, etiket protokolü söylüyor.

DDIA Ch. 1: Reliability gelmesinin nedeni — stateful katmanın (RDS) AWS yönetiminde, stateless katmanın (Fargate) horizontal scale'i kolay; Ch. 3: System of Record (Postgres) ile derived data (S3 mirror) ayrımı net.
DDIA
Ch. 5
Replication

RDS Multi-AZ = Synchronous Leader-Follower

Aynı veri Frankfurt'taki iki farklı Availability Zone'da eşzamanlı tutuluyor. Synchronous replication → write Postgres'e gider, ancak iki AZ'de de yazıldıktan sonra ACK döner. Leader düşerse otomatik failover (~30sn). Bu DDIA'nın tarif ettiği "single-leader synchronous replication" pattern'inin AWS-managed hâli.

Diyagram 2 — Talep Oluşturma Sequence (Üretici → Tesis)

Aynı senaryo zaman ekseninde. Yukarıdan aşağıya zaman akıyor. Her dikey çizgi bir bileşen, yataylar mesajlar.

DDIA Ch. 7: Transactions — RLS check + INSERT + audit + idempotency aynı atomic transaction. Birinin başarısız olması hepsini geri alıyor. Ch. 11: Stream processing — event yayını commit'ten sonra oluyor; eventually consistent fan-out.
DDIA
Ch. 7
Transactions

RLS + Insert + Audit Aynı Transaction'da

Üç şey atomically commit ediliyor: (1) RLS'in policy check'i, (2) tender row insertion, (3) audit_events row insertion (önceki audit row'unun hash'iyle zincirli). Birinden biri başarısız olsa hepsi geri alınıyor. Audit "best effort" değil, transactional invariant — bir tender oluşmuşsa kesin audit'i de var. Bu DDIA'nın tarif ettiği "ACID transactions for invariants" pattern'i.

DDIA
Ch. 11
Stream Processing

Post-Commit Event Emission

Event Bridge'e tender.created event'i ancak transaction commit olduktan sonra atılıyor — Postgres'in transactional outbox pattern'i ile (tablodaki yeni row'u trigger okuyup EventBridge'e gönderiyor). Niye önemli: rollback olursa event de hiç atılmamış oluyor. Bid alıcılar yanlış bilgi almıyor. Bu "exactly-once-into-the-world" semantics'in pratik karşılığı.

Diyagram 3 — Auction Lifecycle State Machine

İhale durum makinesi. Para'ya dokunan kritik akış. Hangi durum geçişlerinin server-authoritative olduğu, hangi event'lerin atıldığı görülüyor.

DDIA Ch. 8: "Don't trust the clock" — closes_at deadline'ı sadece server-side scheduler tetikler. Client'ın saati hiçbir state geçişinde otorite değil. Ch. 12: Event log her geçişin kaydı; rebuild edilebilir.
DDIA
Ch. 8
Untrustworthy Clocks

Server-Authoritative Lifecycle

Distributed sistemde client clock'u manipüle edilebilir, network gecikmeli, NTP drift'li. DDIA Ch. 8 bu yüzden "monotonic clocks for measuring elapsed time, time-of-day clocks for points in time" diye uyarıyor. Bizde her state transition (özellikle CLOSING geçişi) sadece EventBridge Scheduler ile tetiklenir — closes_at timestamp'ı Postgres'e yazıldığı an dondurulmuş, server'ın kendi monotonic clock'una göre evaluate ediliyor. Last-minute sniping de aynı sebepten soft-close pattern'iyle absorbe ediliyor.

Diyagram 4 — System of Record vs Derived Data (Audit + Event Log)

DDIA Ch. 12'nin merkez fikri: tek bir veri yerini kaynak olarak belirle, geri kalanı ondan derive et. Audit ve event akışında bu prensibi uyguluyoruz.

DDIA Ch. 12: "Deriving data" — SoR tek, derived'lar çok. Her derived store SoR'dan rebuild edilebilir. Bu Lego mimarinin teorik temeli.
DDIA
Ch. 12
Derived Data

Tek SoR, Çok Derived — Yeni Modül = Yeni Consumer

Postgres single source of truth. EventBridge'i okuyarak yeni derived store'lar oluşturuluyor: Phase 2 IoT için TimescaleDB hypertable'ı (events → time-series); Phase 2 search için OpenSearch index (events → full-text); Phase 3 ML için S3 data lake (events → Parquet). Hepsi Phase 1 kodunu değiştirmeden ekleniyor. Bu pattern DDIA'nın "log-based architecture" olarak tarif ettiği şey, ve senin "Lego parçası gibi eklenecek" çerçevenin teorik karşılığı.

Vurgulanması Gereken İki Ek Prensip

DDIA
Ch. 8
Idempotency

Her Mutasyonda Idempotency Key

Distributed sistemde "exactly-once delivery" mit. Onun yerine "at-least-once delivery + idempotent receivers". Her mutasyon endpoint'i bir Idempotency-Key header'ı bekliyor; aynı key ikinci kez gelirse aynı response dönüyor, side effect yeniden çalışmıyor. Phase 1'de "nice to have" değil — escrow'a girince absolute zorunluluk. Aynı ödeme isteği iki kez tetiklenirse ödeme iki kez çekilir, felaket. Disiplini Phase 1'de oturtuyoruz, Phase 2'de hediye olarak hazır geliyor.

DDIA
Ch. 1
Evolvability

Çift-Katmanlı RLS = Schema-Level Maintainability

Multi-tenant isolation iki boyutlu: (1) org membership (bir tenant'ın verisi başka tenant'a sızmaz), (2) intra-org role (Muhasebe finans tablosunu görür, Operasyon görmez). Kural Postgres'in içinde — uygulama kodu RLS'i bypass edemez, çünkü Postgres bağlantısının kendisi düşük yetkili. Tüm policy'ler ortak current_org_id() ve user_has_role_in_org() SQL function'ları üzerinden yazıldığı için yeni tablo eklemek tek satır policy = "USING (org_id = current_org_id())". CI'da explicit RLS test suite ile cross-tenant data leak baştan testleniyor — bu B2B SaaS'in en sessiz ama en kritik disiplini, ve DDIA'nın "evolvability" prensibinin somut karşılığı: yeni feature eklerken security boundary'lerini düşünmek zorunda kalmıyorsun, zaten policy pattern'i hazır.

08. Yapay Zeka Kullanımı

AI'ı velocity için kullanırım, correctness için değil — ayrımı net tutuyorum. "Daha hızlı yazılım" ile "daha güvenli yazılım" farklı eksenler.

AI'dan Faydalandığım Yerler

  • Planlama (en değerli) — mimari brainstorm, trade-off analizi, schema tasarım kritiği
  • Debugging — log triage, hata pattern tanıma
  • Coding — CRUD scaffold, form/tablo bileşenleri, migration draft'ları, IaC draft'ları
  • Testing — özellikle edge case üretimi
  • Dokümantasyon — API docs, runbook taslakları

Cursor + Claude Code günlük akışta sürekli açık.

Manuel Kontrolü Asla Bırakmadığım Yerler

  • RLS politikaları — güvenlik sınırları, threat-modeled
  • Production migration'lar — draft AI'dan, apply manuel
  • Auth / session kodu — yüzey çok geniş, hata maliyeti yüksek
  • Audit log şeması + append-only zorlamalar — yasal kanıt
  • Escrow / ödeme tarafı — para'ya dokunan kod, AI auto-apply yok

Bunlar KVKK + para + kimliğe dokunan yüzeyler. Yanlışın maliyeti AI'ın kazandırdığı zamandan kat kat fazla.

Somut guardrail: AI üretimli kod CI gate'lerinden geçmeden merge olmaz — typecheck, lint, RLS policy testleri, schema-drift kontrolü, secret scan. Hızı AI'dan, doğruluğu insandan.

11. Sözlük (Hızlı Referans)

Burada sürekli kullandığım terimler. Konuşurken referans olarak duruyor.

Hot Path
Kullanıcının her tıklamada hissettiği interactive akış. Latency UX'i direkt etkiliyor. Login, form submit, canlı bid, dashboard render.
Cold Path
Arka planda çalışan, kullanıcı beklemeyen iş. Maliyet ve dayanıklılık önemli. PDF üretim, audit mirror, backup, log archival.
DPA
Data Processing Agreement. Müşteriyle imzalanan "verinizi şu şekilde işliyoruz" sözleşmesi. KVKK için müşteri = veri sorumlusu, biz = veri işleyen.
SCC
Standard Contractual Clauses. AWS gibi yurt dışı sağlayıcıyla "veri yurt dışında ama korumalı" pozisyonu kuran standart sözleşme şablonu.
RLS
Row-Level Security. Postgres'in native özelliği — kullanıcının hangi satırları görebileceği veritabanı seviyesinde tanımlanır, uygulama kodu bypass edemez.
Multi-Tenant
Tek uygulamanın birden fazla müşteri (org) verisini aynı veritabanında tutması, ama mantıksal olarak izole etmesi. RLS ile yapıyoruz.
Multi-AZ
Multiple Availability Zones. Aynı verinin farklı binalarda kopyalanması — bir bina kapansa diğeri devralıyor.
Local Zone
AWS'in compute uzantısı, parent region'a bağlı. İstanbul'da var ama full region değil — RDS, S3, EventBridge gibi servisler parent region'da kalıyor.
Idempotency
Aynı isteği birden fazla göndermek aynı sonucu vermesi. Her mutasyona idempotency key veriyoruz, retry'lar duplikat üretmiyor.
EventBridge
AWS'in managed event bus'ı. Modüller event yayar, diğer modüller pattern'e göre dinler. Bizim "Lego mimarinin" omurgası.
Append-Only
Sadece ekleme yapılabilen, update/delete kapalı tablo. Audit log için. Postgres RLS ile zorlanıyor + S3 Object Lock ile mirror'lanıyor.
Server-Authoritative
Önemli kararlar (auction kapanma vs.) sunucuda alınır, client'a güvenilmez. Saat manipülasyonu, network gecikmesi etkilemez.
Soft-Close
İhalede son saniyelerde gelen bid süreyi otomatik uzatır — sniping'i (son saniye yüksek teklif) önler.
Vendor Lock-in
Bir sağlayıcıya öyle bağlı kalmak ki çıkış maliyeti baskıcı olur. Supabase OSS self-host = vendor biziz, lock-in yok.
Modüler Monolith
Tek deployment ama içeride modüller bağımsız. Mikroservis karmaşıklığı yok, ama büyüme yolu açık.
Infrastructure as Code (IaC)
Altyapı kod olarak yazılır (CDK / Terraform). Console'da elle tıklamak yerine reproducible, version-controlled, code-review'lanabilir.

Bu doküman sohbet için hazırlanmıştır. İçerik tartışmaya açık — fazların scope'u, escrow sağlayıcı, carrier surface'i, AI vendor seçimi gibi noktalar yarın birlikte netleştirilecek.
