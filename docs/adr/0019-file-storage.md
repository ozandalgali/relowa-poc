# ADR-0019 — File Storage Strategy

**Status:** Accepted
**Date:** 2026-05-14
**Decision-makers:** Ozan (lead)

## Context

The platform handles several distinct file categories with different security, retention, and access requirements:

1. **Tender photos** — uploaded by producers, viewable by recyclers; fuel the AI scan (Greyparrot).
2. **Registration documents** — Çevre Lisansı, K1, vergi levhası, etc. (PRD-0009). Sensitive PII; staff-only access.
3. **e-fatura PDFs** — generated by Nilvera/Foriba; long retention required by Turkish law.
4. **Audit archive (S3 WORM)** — daily JSON-Lines export of `audit_events` (ADR-0008).
5. **ESG certificates** — PDFs anchored on Arbitrum; user-downloadable.
6. **Static assets** — logos, public landing images, etc.
7. **User avatars + org logos** — small, public-readable.
8. **Backups** — RDS snapshots (managed by AWS); separate concern from this ADR.

Without a unified strategy:
- Buckets proliferate without consistent encryption/retention/access policy.
- File uploads bypass scanning → malware risk.
- KVKK retention obligations get violated.
- Cost grows untracked.
- Direct S3 access from frontend leaks credentials or allows abuse.

## Decision

We adopt a **5-bucket layout** in S3, each with specific encryption, access pattern, retention, and lifecycle. All uploads go through **presigned URLs**; downloads go through **presigned URLs or CloudFront**. Every uploaded file is virus-scanned before being marked usable.

### 1. Bucket layout

| Bucket | Purpose | Encryption | Access | Retention |
|---|---|---|---|---|
| `relowa-tender-photos-prod` | Producer-uploaded tender images | SSE-KMS | Presigned (operator + recycler in same auction) | 2 years after tender settled; KVKK m.4 |
| `relowa-org-documents-prod` | Çevre Lisansı, K1, vergi levhası | SSE-KMS (separate CMK) | Presigned (staff + uploading org) | 10 years per tax regulation |
| `relowa-efatura-prod` | Issued e-fatura PDFs | SSE-KMS | Presigned (involved orgs only) | 10 years per VUK m.253 |
| `relowa-audit-archive-prod` | Daily audit_events JSONL exports + Merkle proofs | SSE-KMS + Object Lock (Compliance mode) | Read-only by staff; immutable | 10 years (Object Lock) |
| `relowa-public-assets-prod` | Logos, landing images, static assets | SSE-S3 | CloudFront-cached, public | Indefinite |

Plus 5 mirrored `*-dev` buckets for dev environment.

**Why separate buckets:**
- Different KMS keys per sensitivity (separate compromise blast radius).
- Different lifecycle policies (e.g. tender-photos auto-expire; audit-archive immutable).
- Different IAM policies (staff sees docs; never sees tender photos by default).
- Clear billing breakdown.

### 2. Encryption

All buckets except `public-assets`:
- SSE-KMS with **separate Customer Managed Keys** per bucket (CMK rotation 365 days, auto via Secrets Manager).
- Bucket policy denies all `s3:*` unless `aws:SecureTransport = true`.
- Versioning enabled.
- Block public access ON (no exceptions; presigned URLs work without public access).

`public-assets`:
- SSE-S3 (server-side AES-256, AWS-managed).
- Public-read via CloudFront origin; bucket itself private.
- CloudFront cache TTL: 24h for assets, 1h for logos.

### 3. Upload flow (presigned PUT)

```
1. Client requests upload URL:
     POST /api/uploads/request
     body: { category, filename, content_type, size }

2. API validates:
     - User has permission (via RLS for the target — e.g. tender owner)
     - Category exists in policy table
     - content_type matches category whitelist (e.g. image/* for tender-photos)
     - size <= category limit (e.g. 10MB for tender-photos)
     - User isn't rate-limited (ADR-0022)

3. API generates:
     - s3_key = '<category>/<org_id>/<entity_type>/<entity_id>/<uuid>.<ext>'
     - Presigned PUT URL (expires in 15 min)
     - Inserts uploads row with status='pending'

4. Client PUTs file directly to S3 using the presigned URL.

5. S3 → EventBridge: PutObject event in 'relowa-*-prod' bucket
   → triggers virus-scan Lambda

6. virus-scan Lambda:
     - Reads file from S3
     - Scans with ClamAV (Lambda layer)
     - On clean: UPDATE uploads SET status='clean'
     - On infected: DELETE object, UPDATE uploads SET status='infected', alert security
     - On scan failure (timeout, oversized): UPDATE status='scan_failed', alert

7. Client polls GET /api/uploads/:id until status='clean'

8. Client commits the upload:
     PATCH /tenders/:id  body: { photo_keys: [<upload.s3_key>] }
     - API validates the upload is 'clean' and belongs to this user
     - Updates the entity (tender.photo_keys)
     - audit_events row written
```

`uploads` table:

```sql
CREATE TYPE upload_status AS ENUM (
  'pending',       -- presigned URL issued
  'uploading',     -- not used; clients PUT directly to S3
  'scanning',      -- in virus scan
  'clean',         -- usable
  'infected',      -- malware detected; deleted
  'scan_failed',   -- scan timed out / file too big / unrecognized format
  'rejected',      -- failed content-type / size validation
  'expired'        -- presigned URL expired without successful upload
);

CREATE TABLE uploads (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID NOT NULL REFERENCES users(id),
  org_id        UUID NOT NULL REFERENCES organizations(id),
  category      TEXT NOT NULL,                          -- 'tender_photo' | 'org_document' | etc
  bucket        TEXT NOT NULL,
  s3_key        TEXT NOT NULL,
  content_type  TEXT NOT NULL,
  size_bytes    BIGINT,                                  -- known after upload
  status        upload_status NOT NULL DEFAULT 'pending',
  status_reason TEXT,
  scan_result   jsonb,                                   -- clamav details
  expires_at    TIMESTAMPTZ NOT NULL,
  uploaded_at   TIMESTAMPTZ,
  scanned_at    TIMESTAMPTZ,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX uploads_user_idx ON uploads(user_id, created_at DESC);
CREATE INDEX uploads_scanning_idx ON uploads(status) WHERE status IN ('pending', 'scanning');
```

**RLS:** users see only their own uploads.

### 4. Download flow (presigned GET)

```
1. Client requests download:
     GET /api/files/:s3_key/url

2. API validates:
     - User has read permission for the entity referencing this key
     - The key is 'clean' (not infected)
     - Apply category-specific authorization (e.g. tender-photo readable by tender.org + bidder orgs)

3. API generates presigned GET URL (expires in 5 min for sensitive; 1h for public)

4. Client follows URL to download.

5. Audit:
     - For sensitive categories (org-documents, e-fatura), insert audit_events row with action='file.accessed'
     - For tender photos: no per-download audit (too noisy)
```

CloudFront fronts `public-assets`; presigned URLs front everything else (CloudFront's signed URLs are an option but presigned S3 is simpler).

### 5. Virus scanning

ClamAV via Lambda Layer:

- Lambda timeout: 60s (most files scan in <5s)
- Memory: 2GB (ClamAV's signature DB is ~250MB)
- Triggered by S3 EventBridge events on `*-prod` buckets except `public-assets`
- ClamAV signature DB refreshed daily via separate Lambda
- On malware: file deleted, security alert fires (ADR-0020)
- On scan failure for oversized files: rejected at presigned-URL request (size limit pre-enforced)

Files larger than `MAX_SCAN_SIZE` (default 100MB):
- Currently rejected. P1 doesn't support large files.
- Phase 2: chunked scan or skip-with-flag for video uploads if needed.

### 6. Lifecycle and retention

Per-bucket lifecycle:

```
relowa-tender-photos-prod:
  - Transition to S3 Standard-IA after 90 days
  - Transition to Glacier Instant Retrieval after 1 year
  - Delete 2 years after the referencing tender.settled_at
  - Implementation: tag objects with `tender_settled_at`; lifecycle reads tag

relowa-org-documents-prod:
  - Standard for 1 year
  - Glacier IR thereafter
  - Delete 10 years after org.deleted_at (or never if org active)

relowa-efatura-prod:
  - Standard 30 days
  - IA 1 year, Glacier IR thereafter
  - Object Lock Compliance mode: 10 years (cannot delete)

relowa-audit-archive-prod:
  - Object Lock Compliance mode: 10 years
  - Glacier Deep Archive after 1 year (read still possible, just slow)

relowa-public-assets-prod:
  - No transition (small, hot, served via CloudFront)
```

### 7. KVKK considerations

- **Data subject erasure (KVKK m.13):** when an org requests deletion, soft-delete `users.deleted_at`. Files in `tender-photos` and `org-documents` are NOT immediately deleted — they sit until lifecycle policy expires them, OR a `compliance-officer` can trigger immediate deletion via admin tool (audit-logged).
- **Audit archive immutability** trumps deletion requests — anchored hashes are legally evidence and cannot be erased. The KVKK response template includes this disclosure.
- **Data export (KVKK m.13):** the export ZIP contains:
  - All `tenders`, `bids`, `audit_events` rows for the user.
  - All `uploads` rows + signed URLs to the actual S3 objects (24h validity).
  - All `notifications` rows.
  - Hash of the user's audit chain entries for independent verification.

### 8. Cost model

Pilot scale (50 producers × 5 tenders/mo × 5 photos = 1250 photos/mo at avg 1MB):

| Bucket | Storage growth | Monthly cost |
|---|---|---|
| `tender-photos` | ~1.5 GB/mo | $0.04 (Standard) → $0.01 (IA after 90d) |
| `org-documents` | ~50 MB total | $0.01 |
| `e-fatura` | ~100 MB/mo | $0.03 |
| `audit-archive` | ~100 MB/mo | $0.03 (Standard) → $0.001 (Glacier DA) |
| `public-assets` | ~500 MB total | $0.02 |
| **Total storage** | | **~$0.15/mo growing** |
| KMS calls | ~5k requests | $0.05 |
| CloudFront | ~5 GB egress | $0.50 |
| Lambda (virus scan) | ~1500 invocations | $0.02 |
| **Total** | | **~$1/month** at pilot scale |

At 10x pilot scale: ~$10/mo. Still negligible.

### 9. Access patterns

| Category | Who reads | How |
|---|---|---|
| tender-photo | Producer (owner) + bidders (recyclers in active auction) | Presigned GET via API |
| org-document | super_admin, account_manager (assigned), compliance_officer | Presigned GET via admin API; audit logged |
| e-fatura | Both parties on invoice | Presigned GET via /finans/faturalar |
| audit-archive | super_admin, compliance_officer | Presigned GET via admin API; audit logged |
| public-asset | Anyone | CloudFront cached, no auth |

### 10. Frontend integration

The component inventory (`docs/frontend/component-inventory.md`) has `FileDropzone` (Layer 1). Implementation:

```tsx
const upload = useUpload({ category: 'tender_photo', tenderId });
<FileDropzone
  onFiles={(files) => upload.start(files)}
  status={upload.status}              // 'idle' | 'requesting' | 'uploading' | 'scanning' | 'clean' | 'error'
  errorMessage={upload.error}
/>
```

The `useUpload` hook handles:
1. POST /api/uploads/request → get presigned URL.
2. PUT to S3 with progress events.
3. Poll GET /api/uploads/:id until clean.
4. Surface status to UI.

The Figma AI-scan animation overlays during `scanning` status.

## Consequences

### Positive

- **Single discipline** for every file in the system.
- **Virus scan is mandatory** — no file bypasses it.
- **Presigned URLs** mean no AWS creds in frontend or backend (API generates short-lived URLs).
- **Retention is automated** via lifecycle policies; KVKK obligations met by configuration, not by remembering to delete.
- **Audit archive is immutable** — Object Lock is legal evidence.
- **Cost is observable** per category via bucket-level CloudWatch metrics.
- **CloudFront for public assets** is cheap + fast.

### Negative

- **5 buckets per env = 10 total** is operational overhead. Mitigated by Terraform module.
- **ClamAV signature DB stale** is a real risk; daily refresh Lambda mitigates.
- **Presigned URLs grant time-bound full access** — anyone with the URL within 5 min can download. Mitigated by short expiry + IP restrictions where feasible.
- **Object Lock cannot be undone** — accidentally writing the wrong file to `audit-archive` is permanent. Mitigated by writing only the lambda; humans never PutObject there directly.

## Future plans

- **Image resizing / thumbnails** — Lambda on tender-photo upload generates 200px + 800px versions. Phase 2.
- **CloudFront signed URLs** for sensitive content if presigned S3 latency becomes a problem. Phase 2.
- **Antivirus alternative** (e.g. Bitdefender for Lambda) if ClamAV miss-rate is too high. Phase 2.
- **Multi-region replication** for `audit-archive` to ap-southeast / us-east as a DR copy. Phase 3.
- **Document OCR** on org-documents — extract Çevre Lisansı number, validate format automatically. Phase 2.
- **Video uploads** if shipment proof-of-delivery becomes video. Phase 2.
- **Customer-managed encryption keys** for enterprise tenants. Phase 3.

## Alternatives considered

| Option | Rejected because |
|---|---|
| Single bucket with prefixes | Loses per-category lifecycle + IAM granularity. |
| Direct S3 PUT from browser (no presigned URL) | Exposes credentials or requires public bucket; security hole. |
| Skip virus scan in P1 | Tender photos from random producers carry malware risk. Pilot orgs trust us; we must earn it. |
| Use S3 alone for all files including public assets | Egress cost dominates; CloudFront for public makes sense. |
| Cloudflare R2 instead of S3 | Different IAM model; KVKK paperwork; ecosystem fit worse with AWS-native rest of stack. |
| Glacier-only for audit archive | Retrieval cost + latency too high for occasional compliance lookups. Standard → Glacier transition is the compromise. |
| Bucket per tenant | Operationally insane at scale; per-bucket policies are fine. |
