# Skill: migration-author

## When to invoke

- Adding a new table.
- Adding a new column to an existing table.
- Adding or modifying RLS policies.
- Adding triggers, functions, or scheduled jobs.
- Modifying enums.

**Do NOT invoke this skill for:** application code, API endpoints, indexes/perf work (that's `db-operator`), or seed data changes that don't alter schema.

## Required reading

Open and read fully before producing any output:

- `AGENTS.md` §2 (RLS as security boundary)
- `packages/db/src/schema.ts` — current Drizzle schema
- `packages/db/src/migrate.ts` — the two-phase migration runner
- `packages/db/src/migrations/` directory listing (don't read every file; skim the latest 2)
- `docs/adr/0001-postgres-as-system-of-record.md`
- `docs/adr/0003-rls-with-jwt-guc-pattern.md`
- `docs/memory/concepts/auth-uid-pattern.md`
- `docs/memory/concepts/multi-tenancy.md`
- The ADR governing the feature you're adding schema for (e.g. ADR-0010 for carrier sub-auction)

## Inputs

- The schema additions implied by the requesting ADR/feature.
- Existing related tables (don't duplicate columns/patterns).

## Outputs

For a typical "add a table" task:

1. **Schema additions** in `packages/db/src/schema.ts`. Use existing patterns (snake_case column names, descriptive enum values, `withTimezone: true` on timestamps).
2. **Drizzle migration** generated by `pnpm db:generate`. Review the SQL output before committing.
3. **Raw SQL side-car** in `packages/db/src/migrations/NNNN_<descriptive-name>.sql` containing:
   - `ALTER TABLE … ENABLE ROW LEVEL SECURITY;`
   - At least one policy per relevant `cmd` (SELECT, INSERT, UPDATE — DELETE only if deletion is allowed)
   - Any triggers (audit, `updated_at`, validation)
   - Publication add if the table needs realtime (`ALTER PUBLICATION supabase_realtime ADD TABLE …`)
4. **Register the side-car** in `packages/db/src/migrate.ts`'s `RAW_SQL_FILES` array.
5. **Hand off** to `rls-test-runner` to add isolation cases.
6. **Notify `doc-keeper`** if the change introduces a new concept or alters an ADR's accuracy.

## Non-negotiables

- ❌ **Never** disable RLS on an existing table. Add a policy that handles the new requirement.
- ❌ **Never** add an UPDATE or DELETE policy on `audit_events` or `admin_audit_log`. The append-only invariant is sacred.
- ❌ **Never** use `auth.uid()` as a column default. It's request-scoped.
- ❌ **Never** write a helper function that touches a table with self-referencing RLS without `SECURITY DEFINER` + explicit `SET search_path = public, pg_temp`.
- ❌ **Never** edit a migration file that's already been applied. Write a new migration instead.
- ❌ **Never** add a table without RLS. The substrate breaks if even one table is unprotected.
- ✅ **Always** name policies descriptively: `<table>_<cmd>_<who>` like `tenders_insert_admin_or_ops`.
- ✅ **Always** prefer `auth.user_org_ids()` over inline subqueries against `org_members`.
- ✅ **Always** include a comment block at the top of the side-car explaining what each policy enforces.

## Verification

```bash
pnpm db:reset                # full rebuild from scratch must succeed
./tests/rls-isolation.sh     # all RLS tests pass
```

If either fails, the task is not done.
